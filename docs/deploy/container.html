Container deployment

Deploy ytdlfin on any Linux host using rootless Podman

Prerequisites

Note: The following must already be in place before following this guide. Installation and configuration of these components is not covered here.
  • Jellyfin — running on the host with its media library accessible at known filesystem paths.
  • Rootless Podman and podman-compose — installed on the host.
  • nginx — configured to terminate TLS and reverse-proxy to ytdlfin.
  • TLS certificate — issued by Let's Encrypt (via certbot) or another trusted CA.
  • PocketID or another OIDC provider — with a client application registered for ytdlfin, yielding a client ID and client secret.

Why rootless Podman

ytdlfin must write downloaded files directly into Jellyfin's media directories. Running the container as root or applying complex ACLs to share access are both undesirable. Rootless Podman with --userns=keep-id (set as userns_mode: keep-id in the compose file) solves this cleanly: the Linux kernel maps the UID and GID of the user who launched Podman straight into the container unchanged. As a result, any file the container process writes to a mounted volume appears on the host owned by that same user — no root required, no ACL changes needed.

This is why the container must be started by (or run as) the same user that Jellyfin runs as. On most systems that user is jellyfin.

Step-by-step deployment

1. Install Podman and podman-compose

On Ubuntu or Debian:

sudo apt update
sudo apt install podman podman-compose

Verify the installation:

podman --version
podman-compose --version

2. Create data and staging directories

Create the directories that will hold the SQLite database and the in-progress download staging area. Both must be owned by the jellyfin user. 

sudo mkdir -p /var/lib/ytdlfin/data /var/lib/ytdlfin/staging
sudo chown -R jellyfin:jellyfin /var/lib/ytdlfin
Tip: For atomic moves (rename instead of copy), STAGING_DIR should be on the same filesystem as your Jellyfin media library. If your library lives on a separate mount, consider placing the staging directory there — for example, /path/to/jellyfin/.ytdlfin-staging.

3. Create a working directory and copy the compose file

Run the following as the jellyfin user. Copy compose.yml from the contrib/ directory of the ytdlfin repository and edit it to match your environment. Pay attention to the # EDIT ME comments. 

sudo -u jellyfin mkdir -p /home/jellyfin/.config/ytdlfin
# Copy from the repo or download from GitHub, then edit
sudo -u jellyfin cp contrib/compose.yml /home/jellyfin/.config/ytdlfin/compose.yml

The full annotated compose file (contrib/compose.yml):

# ytdlfin — rootless Podman deployment
#
# Prerequisites: Podman, podman-compose, Jellyfin, nginx + Let's Encrypt, PocketID (or other OIDC provider)
# Full guide: https://genebean.github.io/ytdlfin/deploy/container.html
#
# Quick start:
#   1. Copy this file and ytdlfin.env.example to a working directory.
#   2. Rename ytdlfin.env.example → ytdlfin.env and fill in the secrets.
#   3. Edit the environment and volume sections below (look for EDIT ME comments).
#   4. Run as the jellyfin user: podman compose up -d

name: ytdlfin

services:
  ytdlfin:
    image: ghcr.io/genebean/ytdlfin:latest
    restart: on-failure:5

    # --userns=keep-id maps the UID/GID of the user running podman (i.e. the
    # jellyfin user) into the container unchanged.  Files written to the
    # mounted volumes will be owned by that same UID on the host, so Jellyfin
    # can read them without any permission changes.
    userns_mode: keep-id

    # Secrets: SECRET_KEY, OIDC_CLIENT_ID, OIDC_CLIENT_SECRET
    env_file: ytdlfin.env

    environment:
      DATA_DIR: /data
      STAGING_DIR: /staging
      PORT: "8001"

      OIDC_ISSUER_URL: https://id.example.com          # EDIT ME
      OIDC_REDIRECT_URI: https://ytdlfin.example.com/auth/callback  # EDIT ME
      ADMIN_GROUP: ytdlfin-admins                       # EDIT ME — must match PocketID group name exactly
      USER_GROUP: ytdlfin-users                         # EDIT ME — must match PocketID group name exactly

      # Must be true: the app expects to sit behind an HTTPS reverse proxy
      # (nginx + Let's Encrypt).  This marks session cookies Secure and enables
      # correct scheme detection from X-Forwarded-Proto.
      HTTPS_ONLY: "true"
      # The IP(s) of your nginx proxy.  "127.0.0.1" is correct when nginx runs
      # on the same host.  Set to "*" only on a fully firewalled LAN.
      TRUSTED_PROXY_IPS: "127.0.0.1"

      LOG_LEVEL: info

    volumes:
      # Persistent data (SQLite database).  Back up this directory with restic.
      - /var/lib/ytdlfin/data:/data

      # Staging area for in-progress downloads.  For atomic moves, this must
      # be on the same filesystem as your Jellyfin library paths below.
      # EDIT ME — ideally something like /path/to/jellyfin/.ytdlfin-staging
      - /var/lib/ytdlfin/staging:/staging

      # Jellyfin media library.  Mount at the EXACT SAME absolute path as on
      # the host so that category paths stored in the database remain valid
      # inside the container.  Add one entry per library root.
      # EDIT ME
      - /path/to/jellyfin/media:/path/to/jellyfin/media

    ports:
      # Bind only to localhost — nginx terminates TLS and proxies here.
      - "127.0.0.1:8001:8001"
Important: Media library volumes must be mounted at the exact same absolute path as on the host. Category paths are stored as absolute paths in the SQLite database; if the path inside the container differs, writes will fail.

4. Create the secrets file

Copy ytdlfin.env.example to ytdlfin.env in the same directory as your compose.yml, then fill in the real values. Set the file mode to 0600 so only the jellyfin user can read it. 

sudo -u jellyfin cp contrib/ytdlfin.env.example /home/jellyfin/.config/ytdlfin/ytdlfin.env
sudo chmod 0600 /home/jellyfin/.config/ytdlfin/ytdlfin.env

Contents of ytdlfin.env.example:

# ytdlfin secrets — copy to ytdlfin.env and fill in real values.
# This file is loaded by podman compose as env_file.
# Keep ytdlfin.env mode 0600 and never commit it.

# 32+ random characters.  Generate with: openssl rand -hex 32
SECRET_KEY=CHANGE_ME

# From your PocketID (or other OIDC provider) client application registration.
OIDC_CLIENT_ID=CHANGE_ME
OIDC_CLIENT_SECRET=CHANGE_ME

Generate a strong secret key:

openssl rand -hex 32

5. Start the stack

Run this as the jellyfin user from the directory containing your compose.yml:

sudo -u jellyfin bash -c 'cd ~/.config/ytdlfin && podman compose up -d'

6. Verify

Tail the logs to confirm the app started cleanly:

sudo -u jellyfin bash -c 'cd ~/.config/ytdlfin && podman compose logs -f'

Then confirm the app is responding on localhost (before nginx is wired up):

curl -s http://127.0.0.1:8001/health

A {"status":"ok"} response confirms the app is running and its database and background worker are healthy. A 503 response includes a detail field identifying which component failed ("db" or "worker").

nginx reverse proxy

Place the following in your nginx configuration (for example, in /etc/nginx/sites-available/ytdlfin). Replace ytdlfin.example.com with your actual domain. After editing, run sudo certbot --nginx -d ytdlfin.example.com to obtain and install the Let's Encrypt certificate, or adapt for your own CA.

Note: The limit_req_zone directive must go in the http {} block (e.g. in /etc/nginx/nginx.conf or a conf.d/ snippet), not inside a server {} block. The limit_req line inside the location block references it by name. 
# In the http {} block — rate-limit the health endpoint so unauthenticated
# bots can't hammer it.  Uptime Kuma polls once per minute; 6r/m gives
# comfortable headroom while still blocking floods.
limit_req_zone $binary_remote_addr zone=ytdlfin_health:1m rate=6r/m;

# HTTP — redirect all traffic to HTTPS
server {
    listen 80;
    listen [::]:80;
    server_name ytdlfin.example.com;

    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS — terminate TLS and proxy to the ytdlfin container
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    server_name ytdlfin.example.com;

    ssl_certificate     /etc/letsencrypt/live/ytdlfin.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ytdlfin.example.com/privkey.pem;
    include             /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam         /etc/letsencrypt/ssl-dhparams.pem;

    # Rate-limit the unauthenticated health endpoint.
    location = /health {
        limit_req zone=ytdlfin_health burst=3 nodelay;
        proxy_pass         http://127.0.0.1:8001;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }

    location / {
        proxy_pass         http://127.0.0.1:8001;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}

Enable the site and reload nginx:

sudo ln -s /etc/nginx/sites-available/ytdlfin /etc/nginx/sites-enabled/ytdlfin
sudo nginx -t && sudo systemctl reload nginx

Health monitoring (Uptime Kuma)

GET /health is an unauthenticated endpoint suitable for uptime monitors. It returns 200 {"status":"ok"} when the app, its database connection, and the background download worker are all healthy. If either the worker task has crashed or the database is unreachable, it returns 503 with a detail field identifying which component failed ("db" or "worker").

In Uptime Kuma, add an HTTP(s) monitor pointed at https://ytdlfin.example.com/health and set the expected status code to 200. No credentials are required.

Auto-start with systemd (optional)

This section is optional. If you prefer another orchestration tool (Ansible, Puppet, etc.) to manage the compose stack, skip it — the stack starts and stops with podman compose up -d and podman compose down regardless.

The repository ships an optional systemd user unit at contrib/ytdlfin.service that starts the compose stack automatically at boot. Because this is a user-level unit (not a system unit), the jellyfin user must have linger enabled so that the unit starts before anyone logs in.

Install and enable the unit (run as the jellyfin user, or prefix with sudo -u jellyfin):

  1. Copy the unit file into the jellyfin user's systemd directory: 
    mkdir -p ~/.config/systemd/user
cp contrib/ytdlfin.service ~/.config/systemd/user/
  2. Edit the WorkingDirectory= line in the unit file to point at the directory containing your compose.yml: 
    WorkingDirectory=%h/.config/ytdlfin
    (%h expands to the user's home directory.)
  3. Enable linger so the user service starts at boot without an interactive login session: 
    sudo loginctl enable-linger jellyfin
  4. Reload the user daemon and enable the unit: 
    systemctl --user daemon-reload
systemctl --user enable --now ytdlfin

Useful commands once the unit is running:

systemctl --user status ytdlfin
systemctl --user restart ytdlfin
journalctl --user -u ytdlfin -f

To trigger an upgrade via systemd (pulls new images then recreates containers): 

systemctl --user reload ytdlfin

Upgrades

Pull the latest image and recreate the container. Run as the jellyfin user from the directory containing your compose.yml:

podman compose pull
podman compose up -d --force-recreate

The image is tagged both as latest and with the version number (for example, ghcr.io/genebean/ytdlfin:1.2.3). Pin to a version tag in compose.yml if you prefer explicit control over when upgrades apply.

Backups

Only DATA_DIR (mounted at /data in the container, defaulting to /var/lib/ytdlfin/data on the host) needs to be included in backups. It contains the SQLite database with all download history and category configuration.

STAGING_DIR (/staging in the container) holds downloads that are currently in progress. It does not need to be backed up — any partial downloads there are discarded and re-queued on restart.

Tip: Back up DATA_DIR with a tool such as restic. A simple daily snapshot of /var/lib/ytdlfin/data is sufficient. SQLite is safe to copy while the app is running as long as you use a method that reads a consistent snapshot (restic, SQLite's .backup command, or a filesystem snapshot).
