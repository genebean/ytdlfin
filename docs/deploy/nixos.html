NixOS deployment

Deploy ytdlfin as a systemd service via the NixOS flake module

Prerequisites

Note: The following must already be in place before following this guide. Installation and configuration of these components is not covered here.
  • NixOS — with flakes enabled.
  • Jellyfin — services.jellyfin.enable = true on the same host.
  • nginx — the NixOS nginx module with recommendedProxySettings = true set at the service level.
  • ACME / Let's Encrypt — security.acme configured with your email address and accepted terms.
  • PocketID or another OIDC provider — with a client application registered for ytdlfin, yielding a client ID and client secret.

Add the flake input

Add ytdlfin as an input in your host flake and include the NixOS module in your system configuration: 

inputs.ytdlfin = {
  url = "github:genebean/ytdlfin";
  inputs.nixpkgs.follows = "nixpkgs";
};

nixosConfigurations.yourhost = nixpkgs.lib.nixosSystem {
  modules = [
    ytdlfin.nixosModules.default
    ./hosts/yourhost/configuration.nix
  ];
};

Configure the service

Add the following to your host's NixOS configuration. The example below reflects a recommended production setup: the service runs as the Jellyfin user, the staging directory sits on the same filesystem as the Jellyfin library, and secrets are managed by sops-nix. 

services.ytdlfin = {
  enable = true;
  user  = config.services.jellyfin.user;
  group = config.services.jellyfin.group;
  stagingDir = "/your/jellyfin/staging/.ytdlfin-staging";
  environmentFile = config.sops.secrets.ytdlfin-env.path;
  settings = {
    oidcIssuerUrl   = "https://id.example.com";
    oidcRedirectUri = "https://ytdlfin.example.com/auth/callback";
    oidcAdminGroup  = "ytdlfin_admins";
    oidcUserGroup   = "ytdlfin_users";
    mediaDirectories = [ "/your/jellyfin/data" ];
  };
};
User and group: Setting user and group to the Jellyfin service account means ytdlfin already has write access to the Jellyfin media library. No ACL changes or additional permissions are needed.
mediaDirectories: Each path listed here is added to the systemd service's ReadWritePaths. The hardened sandbox blocks all other filesystem writes, so ytdlfin cannot write outside these paths.
stagingDir: Place the staging directory on the same filesystem as your Jellyfin library. When a download completes, ytdlfin moves it from stagingDir into the target category directory. If both paths are on the same filesystem that move is an atomic rename; if they are on different filesystems it becomes a full copy.

nginx virtual host

Add the ytdlfin virtual host to your nginx configuration. Because recommendedProxySettings = true is set at the nginx service level, all required proxy headers (Host, X-Real-IP, X-Forwarded-For, X-Forwarded-Proto) are injected automatically. Do not add explicit proxy_set_header lines to the ytdlfin vhost. 

services.nginx = {
  recommendedGzipSettings  = true;
  recommendedOptimisation  = true;
  recommendedProxySettings = true;   # sets Host, X-Real-IP, X-Forwarded-For, X-Forwarded-Proto automatically
  recommendedTlsSettings   = true;

  # Rate-limit the unauthenticated health endpoint so bots can't hammer it.
  # Uptime Kuma polls once per minute; 6r/m gives comfortable headroom.
  # limit_req_zone must live in the http {} block — appendHttpConfig is the
  # right place for it in the NixOS nginx module.
  appendHttpConfig = ''
    limit_req_zone $binary_remote_addr zone=ytdlfin_health:1m rate=6r/m;
  '';

  virtualHosts."ytdlfin.example.com" = {
    enableACME = true;
    acmeRoot   = null;
    forceSSL   = true;
    locations."/".proxyPass = "http://127.0.0.1:${toString config.services.ytdlfin.port}";
    locations."= /health" = {
      proxyPass   = "http://127.0.0.1:${toString config.services.ytdlfin.port}";
      extraConfig = ''
        limit_req zone=ytdlfin_health burst=3 nodelay;
      '';
    };
  };
};
acmeRoot = null: This tells NixOS to use the ACME-managed certificate directory directly rather than a manual webroot challenge path. It is the standard pattern when security.acme is already configured system-wide.

Health monitoring (Uptime Kuma)

GET /health is an unauthenticated endpoint suitable for uptime monitors. It returns 200 {"status":"ok"} when the app, its database connection, and the background download worker are all healthy. If either the worker task has crashed or the database is unreachable, it returns 503 with a detail field identifying which component failed ("db" or "worker").

In Uptime Kuma, add an HTTP(s) monitor pointed at https://ytdlfin.example.com/health and set the expected status code to 200. No credentials are required.

Secret management (sops-nix)

The environmentFile option points to a file containing secrets that must not appear in the Nix store. sops-nix is the recommended approach for NixOS: it decrypts secrets at activation time and places them at a path only the service user can read.

Declare the secret in your sops configuration:

sops.secrets.ytdlfin-env = {
  owner        = config.services.ytdlfin.user;
  restartUnits = [ config.systemd.services.ytdlfin.name ];
};

Any method that produces a file readable by the ytdlfin user works. If you do not use sops-nix, ensure the file mode is 0600 and the owner matches services.ytdlfin.user.

The secret file must contain the following key-value pairs:

SECRET_KEY=<32+ random chars — generate with: openssl rand -hex 32>
OIDC_CLIENT_ID=<client ID from PocketID>
OIDC_CLIENT_SECRET=<client secret from PocketID>

Generate a strong secret key:

openssl rand -hex 32

Module options reference

All options exposed by services.ytdlfin:

Option Default Description
services.ytdlfin.enable false Enable the ytdlfin service.
services.ytdlfin.port 8001 HTTP port the service listens on.
services.ytdlfin.dataDir /var/lib/ytdlfin Directory for ytdlfin.db. This is the only path that needs to be backed up.
services.ytdlfin.stagingDir /var/lib/ytdlfin/staging Temporary directory for in-progress downloads. Override to a path on the same filesystem as the Jellyfin library for atomic moves.
services.ytdlfin.user ytdlfin User to run the service as. Set to config.services.jellyfin.user to share library write access without ACL changes.
services.ytdlfin.group ytdlfin Group to run the service as. Set to config.services.jellyfin.group if running alongside Jellyfin.
services.ytdlfin.environmentFile required Path to a file containing SECRET_KEY, OIDC_CLIENT_ID, and OIDC_CLIENT_SECRET as KEY=VALUE pairs.
services.ytdlfin.settings.oidcIssuerUrl required Base URL of your PocketID (or other OIDC provider) instance, e.g. https://id.example.com.
services.ytdlfin.settings.oidcRedirectUri required Full OIDC callback URL, e.g. https://ytdlfin.example.com/auth/callback.
services.ytdlfin.settings.oidcAdminGroup required PocketID group name whose members receive admin privileges (category management, full history). Must match the group name exactly as configured in PocketID.
services.ytdlfin.settings.oidcUserGroup required PocketID group name whose members have regular user access (submit downloads, view own history). Must match the group name exactly as configured in PocketID.
services.ytdlfin.settings.mediaDirectories [] Jellyfin library root paths. Each path is added to the systemd service's ReadWritePaths; ytdlfin cannot write outside these paths.
services.ytdlfin.settings.logLevel info uvicorn log level. One of debug, info, warning, error.
services.ytdlfin.settings.trustedProxyIps 127.0.0.1 IPs allowed to set X-Forwarded-* headers. The default is correct for same-host nginx. Set to * only for LAN deployments where nginx runs on a different host.

Backups

Only dataDir needs to be included in backups. It contains ytdlfin.db, which holds all download history and category configuration.

stagingDir holds downloads that are currently in progress and is ephemeral. Do not back it up — any partial downloads are discarded and re-queued on restart.

Tip: A simple daily restic snapshot of dataDir is sufficient. SQLite is safe to copy while the app is running as long as you use a method that reads a consistent snapshot (restic, SQLite's .backup command, or a filesystem snapshot). A minimal restic paths entry: 
services.restic.backups.daily = {
  paths = [
    config.services.ytdlfin.dataDir
  ];
};
